# VS-0013 Handoff — Unit tests requiring UI thread failing

**Date:** 2025-12-29  
**Role:** UI Engineer  
**Gate:** C  
**State:** DONE

---

## Summary

Resolved WinUI/XAML unit test failures caused by missing WinUI runtime initialization on the test thread. UI-facing tests now execute on an STA thread with Windows App SDK + WinUI initialization, allowing creation of WinUI controls (e.g., `Button`, `MenuFlyout`) under `dotnet test`.

---

## Changes

### `src/VoiceStudio.App.Tests/UI/UITestMethodAttribute.cs`

- Runs tests on an STA thread.
- Initializes Windows App SDK bootstrap (1.8) for unpackaged test host.
- Creates a `DispatcherQueueController` on the test thread.
- Initializes `WindowsXamlManager.InitializeForCurrentThread()` so WinUI/XAML objects can be activated.

### Tests updated to use UI-thread execution

- `src/VoiceStudio.App.Tests/Services/ContextMenuServiceTests.cs`: converted to `[UITestMethod]` for tests that instantiate WinUI objects.
- Existing UI tests already using `[UITestMethod]` now run successfully after the initialization fixes.

### Supporting fixes for deterministic test behavior

- `src/VoiceStudio.App/Utilities/EnhancedAsyncRelayCommand.cs`: treats missing `canExecute` as "always true" and raises `CanExecuteChanged` when execution starts/ends.
- `src/VoiceStudio.App/ViewModels/GlobalSearchViewModel.cs`: serializes `SearchAsync` to prevent overlapping searches triggered by `OnSearchQueryChanged`.
- `src/VoiceStudio.App.Tests/ViewModels/MockBackendClient.cs` + `GlobalSearchViewModelTests.cs`: added a deterministic blocker hook to validate `IsLoading` transitions.
