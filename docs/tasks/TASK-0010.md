# TASK-0010: Piper/Chatterbox Backend Engine Integration Fix

## Objective

Fix the backend voice cloning synthesis path for **piper** and **chatterbox** engines so that `/api/voice/clone` returns a valid `audio_id` and produces synthesized audio. Currently, these engines are registered and preflight passes, but synthesis does not produce audio.

**Traceability:** Blocked from [TASK-0009](TASK-0009.md) § Tech Debt; [ENGINE_ENGINEER_STATUS_2026-01-27.md](../reports/verification/ENGINE_ENGINEER_STATUS_2026-01-27.md) § TASK-0009 Partial Completion.

## Problem Statement

When running baseline voice proof with `--engine piper` or `--engine chatterbox`:
- Profile is created successfully via `/api/voice/clone`
- Synthesis step returns no `audio_id` (null)
- Baseline proof exits with code 1

**Root cause (updated 2026-01-28):** 
1. **PiperEngine**: Does not support voice cloning - `synthesize()` method does not accept `speaker_wav` parameter. Piper is a standard TTS engine that uses pre-trained voices, not a voice cloning engine. The `/api/voice/clone` route calls `synthesize(speaker_wav=...)` which PiperEngine ignores, resulting in no voice-cloned output.
2. **ChatterboxEngine**: Similar issue - needs verification of `clone_voice()` implementation and whether it properly handles the voice cloning flow.

**Previous root cause (superseded):** The `engine.synthesize()` method in the voice cloning flow does not produce audio output for these engines. The engines are registered in `engine_router` but the synthesis implementation path is incomplete.

## Investigation Summary (2026-01-28)

**Synthesis path traced:** `/api/voice/clone` (voice.py ~2586) uses `clone_voice` if present, else `synthesize()`. For Piper (no `clone_voice`), it calls `engine_instance.synthesize(text=..., speaker_wav=..., output_path=..., calculate_quality=True)`. For Chatterbox, it calls `clone_voice(reference_audio=..., text=..., output_path=..., ...)` which delegates to `synthesize(speaker_wav=reference_audio, ...)`.

**Response building:** `audio_id` is set only when `file_written and output_path` (voice.py ~2912). `file_written` is True when the engine writes to `output_path` or returns an ndarray that voice.py persists to `output_path`. So failure implies either (1) engine returns None, (2) engine throws, or (3) engine not found (get_engine returns None).

**Registration:** Engine router loads from manifests via `load_all_engines("engines")`; fallback manual registration had xtts_v2, chatterbox, tortoise but **not piper**. If manifest load failed or piper was skipped, `get_engine("piper")` would return None.

**Fix applied:** Piper added to manual fallback in `backend/api/routes/voice.py`: import `PiperEngine`, `engine_router.register_engine("piper", PiperEngine)` so piper is available when manifest load fails or omits piper.

**Guard applied:** `/api/voice/clone` now rejects non-cloning engines (HTTP 422) using manifest capabilities (`voice_cloning`/`zero_shot_cloning`). Piper is TTS-only; use `/api/voice/synthesize`.

### Chatterbox verification (Role 4, 2026-01-28)

- **Code path:** `ChatterboxEngine.clone_voice(reference_audio, text, language, emotion, output_path, speed, calculate_quality)` delegates to `synthesize(speaker_wav=reference_audio, ..., output_path=output_path)`. When `output_path` is set, `synthesize()` writes to file and returns `None`. voice.py sets `file_written = os.path.exists(output_path)` and sets `audio_id` when `file_written and output_path`. So the integration contract is correct.
- **Likely root cause for Chatterbox baseline failure:** `voice.py` builds `clone_kwargs` with `enhance_quality`, `use_multi_reference`, `quality_preset`, `prosody_params`. XTTS/Lyrebird/Tortoise `clone_voice` accept these or `**kwargs`; **ChatterboxEngine.clone_voice does not** — it has no `**kwargs` and no `enhance_quality`/`use_multi_reference`/`quality_preset`/`prosody_params`. Calling `engine_instance.clone_voice(**clone_kwargs)` for Chatterbox therefore raises **TypeError: clone_voice() got an unexpected keyword argument 'enhance_quality'** (or similar), so no file is written and no `audio_id`.
- **Fix options (for peer approval):** (1) Add `**kwargs` (or explicit `enhance_quality`, `use_multi_reference`, etc.) to `ChatterboxEngine.clone_voice` and pass through to `synthesize` / ignore; or (2) In `voice.py`, build engine-specific `clone_kwargs` so only supported keys are passed. Option (1) is engine-layer (Role 5); Option (2) is backend (Role 4).

## Implementation (2026-01-28)

1. **Piper (Option B):** In `backend/api/routes/voice.py`, when `engine_id == "piper"` on `/api/voice/clone`, raise `HTTPException(422, detail="Piper does not support voice cloning. Use a voice cloning engine (e.g. xtts_v2, chatterbox).")` so the API returns a clear error instead of attempting synthesis.
2. **Chatterbox (Option 2):** In `voice.py`, before calling `engine_instance.clone_voice(**clone_kwargs)`, filter `clone_kwargs` to only parameters accepted by the engine's `clone_voice` method using `inspect.signature(engine_instance.clone_voice).parameters` (excluding `self`). This avoids `TypeError` from unexpected keyword arguments (e.g. `enhance_quality`, `use_multi_reference`) for engines like Chatterbox that do not accept them.

**Implementation applied (continued):** Engine-specific allowlist added in `voice.py`: `_CLONE_VOICE_ALLOWED_PARAMS["chatterbox"]` restricts Chatterbox to `reference_audio`, `text`, `language`, `emotion`, `output_path`, `speed`, `calculate_quality`. `_CLONE_VOICE_SINGLE_REFERENCE_ONLY` includes `"chatterbox"` so `reference_audio` is normalized to a single path when a list is passed. This ensures Chatterbox never receives unsupported kwargs or a list for `reference_audio`.

**Required proofs:** Re-run `python scripts/baseline_voice_workflow_proof.py --engine chatterbox` when backend and venv are available (and chatterbox-tts installed); Piper baseline proof is expected to fail with 422 (by design).
